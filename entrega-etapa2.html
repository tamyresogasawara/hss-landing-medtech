Este documento acompanha a entrega da Etapa 2 do MedTech Challenge: a transformação do protótipo em um site funcional, publicado online, com formulário de captação de lead operacional, acessibilidade reforçada e responsividade completa. O site está disponível em produção via GitHub Pages e o código-fonte é público.

1. O que evoluiu da Etapa 1 para a Etapa 2

A Etapa 1 entregou o protótipo conceitual: estrutura, identidade visual e copy. A Etapa 2 transforma isso em um produto navegável, real e funcional. As mudanças concentraram-se em quatro frentes:

Formulário operacional

Integração com Formspree para envio real de leads, com loading state, mensagens inline de sucesso/erro e proteção anti-spam por honeypot.

Acessibilidade reforçada

Skip link, focus styles explícitos, suporte a prefers-reduced-motion, ARIA completo no formulário e validação per-field anunciada por leitor de tela.

Polimento de produção

Favicon SVG inline, meta tags Open Graph e Twitter Card para compartilhamento, theme-color para a barra do navegador no mobile.

UX do envio

O usuário recebe feedback em tempo real: erros somem conforme corrige, botão entra em estado de carregamento e o foco se move para o título do formulário ao concluir.

O que se manteve

A identidade visual (paleta branca/azul-petróleo/verde-menta, tipografia Plus Jakarta Sans + Inter), a estrutura de oito seções e o copy persuasivo da Etapa 1 foram mantidos integralmente. Houve apenas refinos pontuais para melhorar a hierarquia do formulário e a clareza dos campos.

2. Site publicado — acesso e infraestrutura

O site está hospedado no GitHub Pages, com HTTPS forçado e CDN global da própria GitHub. O código-fonte é público no GitHub, permitindo auditoria e reprodutibilidade.

Links oficiais

3. Formulário de captação de lead operacional

O formulário é totalmente funcional. Cada submissão válida é enviada via fetch para o endpoint do Formspree, que armazena o lead, dispara notificação por email e disponibiliza o registro no painel administrativo.

Fluxo do envio

1Usuário preenche o formulário
→
2Validação client-side em PT-BR
→
3POST para Formspree
→
4Email + dashboard
→
5Confirmação inline

Campos coletados

  • Nome completo (obrigatório)
  • Cargo (opcional)
  • E-mail corporativo (obrigatório, validado como email)
  • Telefone (obrigatório)
  • Nome da clínica (obrigatório)
  • Quantidade de profissionais (select com 5 faixas)
  • Mensagem livre (opcional, textarea)
  • Consentimento LGPD (checkbox obrigatório)

Segurança e qualidade dos leads

  • Honeypot anti-spam: campo _gotcha invisível ao usuário; bots que preenchem todos os campos são rejeitados pelo Formspree antes de chegarem como lead.
  • Validação dupla: HTML5 nativa + lógica em JavaScript com mensagens em português.
  • Consentimento explícito: checkbox obrigatório linkando à Política de Privacidade, em conformidade com a LGPD.
  • Atributo autocomplete: aplicado em todos os campos para reduzir fricção e permitir preenchimento automático em compliance com WCAG.

Estados de UX

Loading

Botão recebe aria-busy="true", exibe spinner e fica desabilitado durante o envio.

Sucesso

Mensagem verde inline; formulário é resetado; foco move para o título (anunciado por leitor de tela).

Erro de validação

Erros aparecem abaixo de cada campo inválido, com aria-invalid e contraste AA.

Erro de rede

Mensagem clara solicitando nova tentativa, sem perder os dados preenchidos.

4. Acessibilidade implementada

A acessibilidade foi tratada como critério de primeira classe, não como item de checklist. O objetivo é que pessoas usuárias de teclado, leitor de tela ou com sensibilidade a movimento tenham experiência equivalente à de quem usa mouse e enxerga normalmente.

Recursos implementados

Skip link

Primeiro elemento focável da página. Permite pular o cabeçalho e ir direto ao conteúdo principal — essencial para usuários de teclado e leitor de tela.

Focus visible explícito

Todos os elementos interativos têm :focus-visible com outline azul de 3px e offset, sem depender do estilo padrão do navegador.

HTML semântico

Uso consistente de header, nav, main, section, article e footer para que tecnologias assistivas entendam a estrutura.

Hierarquia de headings

Apenas um h1 por página, seguido de h2, h3 e h4 em ordem lógica e sem pular níveis.

ARIA no formulário

Cada campo obrigatório tem aria-describedby apontando para um elemento de erro com aria-live="polite"; aria-invalid é aplicado dinamicamente.

Status global do envio

Container com role="status" e aria-live="polite" anuncia mudanças (envio em andamento, sucesso, erro) sem interromper o usuário.

prefers-reduced-motion

Usuários que pedem movimento reduzido têm animações e scroll-behavior automaticamente desligados via media query.

Atributo lang

Atributo lang="pt-BR" no html, permitindo que leitores de tela usem a pronúncia correta.

Autocomplete

Campos do formulário usam autocomplete apropriado (name, email, tel, organization), reduzindo carga cognitiva e atendendo WCAG 1.3.5.

Contraste de cores

Todas as combinações texto/fundo respeitam WCAG AA: ratio mínimo de 4.5:1 para texto normal e 3:1 para texto grande.

SVGs decorativos ocultos

Ícones puramente visuais marcados com aria-hidden="true", evitando ruído para usuários de leitor de tela.

Foco programático no sucesso

Após envio bem-sucedido, o foco move para o título do formulário (com tabindex="-1"), anunciando a mudança de estado.

Como validar

Execute uma auditoria Lighthouse no Chrome (DevTools → aba Lighthouse → marcar "Accessibility" → "Analyze page load"). O alvo é nota ≥ 95 na categoria Accessibility. A auditoria também aponta automaticamente potenciais problemas remanescentes para refinamento contínuo.

5. Responsividade

O layout foi pensado em abordagem mobile-first: o estilo base é otimizado para tela pequena e os breakpoints adicionam complexidade conforme o espaço disponível. Os textos usam unidades fluidas (clamp()) para escalar suavemente entre tamanhos de tela.

Breakpoints

Largura Comportamento
≥ 1025px Layout completo: hero em duas colunas, grids de 3-4 colunas, menu horizontal.
881px – 1024px Diferenciais reduzem de 4 → 2 colunas, mantendo demais layouts.
721px – 880px Hero, soluções, funcionalidades e públicos colapsam para coluna única; rodapé reorganiza-se.
481px – 720px Menu hamburguer ativa-se; CTAs secundários do header escondem; formulário vira coluna única.
≤ 480px Diferenciais ficam em uma coluna; botões do hero ficam full-width; rodapé empilha.

Técnicas usadas

  • CSS Grid e Flexbox: layouts adaptativos sem media queries para a maioria dos casos.
  • clamp(): tipografia e espaçamentos fluidos entre min e max, com base no viewport.
  • Container fluido: max-width: 1200px com padding lateral responsivo via clamp().
  • Imagens responsivas: SVG e CSS-art escalam por natureza, sem dependência de imagens raster.
  • Menu mobile: ativado via JavaScript mínimo (~10 linhas), com gestão de aria-expanded.

Cobertura testada

Testado em Chrome, Safari e Firefox em três tamanhos de viewport: mobile (375px, iPhone), tablet (768px, iPad) e desktop (1440px). Sem quebras de layout ou overflow horizontal em nenhum tamanho.

7. Decisões técnicas

Stack: HTML5 semântico + CSS puro + JavaScript vanilla

Manteve-se a escolha da Etapa 1 — HTML/CSS/JS puros, sem frameworks. Justificativas:

  • Performance: primeira renderização em milissegundos, sem JavaScript bloqueante crítico.
  • Confiabilidade: sem build step, sem dependências quebrando, sem incompatibilidades de versão.
  • Manutenibilidade: qualquer desenvolvedor abre e contribui sem instalar nada.
  • SEO: conteúdo HTML totalmente renderizado server-side por padrão.

Organização de CSS

Quatro arquivos modulares, cada um com responsabilidade clara:

  • reset.css — normalização cross-browser
  • variables.css — design tokens via CSS Custom Properties (cores, fontes, espaçamentos)
  • base.css — tipografia, layout e recursos de acessibilidade (skip link, focus, reduced-motion)
  • components.css — header, hero, cards, formulário (incluindo estados de erro/sucesso/loading) e footer

JavaScript

Inline no HTML, sem dependências externas. Aproximadamente 110 linhas divididas em duas IIFEs isoladas: toggle de menu mobile e handler do formulário com integração Formspree. Total transferido: menos de 4 KB minificado.

Integração com Formspree

Optou-se pelo Formspree pela maturidade, simplicidade e tier gratuito generoso (50 envios/mês). Alternativas avaliadas:

  • Web3Forms — descartado: menos popular, menos garantia de longevidade.
  • EmailJS — descartado: expõe credencial pública no frontend.
  • Google Apps Script + Sheets — descartado para esta etapa: mais setup; pode ser reconsiderado em fases futuras se houver necessidade de armazenamento estruturado.
  • Backend próprio — overkill para o escopo atual; introduz custo e ponto único de falha.

SEO e compartilhamento

  • Meta description otimizada com palavras-chave do nicho (gestão de clínicas, agenda, prontuário).
  • Open Graph + Twitter Card para preview rico em redes sociais.
  • Favicon SVG inline via data URL — economiza uma requisição HTTP.
  • theme-color meta tag para a barra do navegador no mobile.
